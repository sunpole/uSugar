# Техническая документация: Telegram-бот для семейного мониторинга сахара (Libre2) — версия 1.0.0+

## 1. Общее описание

Бот предназначен для семейной группы (родители+ребёнок с диабетом 1‑го типа).  
**Основные возможности:**

- **Распознавание скриншотов** из приложения Libre2 (уровень глюкозы, mmol/L или mg/dL) с **тройным гибридным OCR** и проверкой согласованности результатов.
- **Расчёт доз инсулина** на еду, на коррекцию и их комбинацию с учётом персональных коэффициентов.
- **Приём и обработка списка углеводов** по ключевым фразам («какой укол», «сколько укола», «посчитай укол») – расчёт ХЕ и дозы.
- **Ручной ввод** замеров сахара, углеводов, ХЕ, инъекций (короткий и длинный инсулин), заметок.
- **Журнал замеров** – вывод за день/неделю/месяц, при необходимости ссылка на файл (CSV).
- **Напоминания** (проверка сахара, укол длинного инсулина, контроль через 2 часа после еды).
- Работает **в личных сообщениях и в групповом чате** (требуются права администратора).

**Целевые версии:**  
- **1.0.0** – полнофункциональный бот на домашнем ПК/локальном сервере.  
- **2.0.0** – деплой на выделенный хостинг 24/7, персистентность состояний, webhook.  
- **2.0.1** – интеграция **DeepSeek API** для расширенной аналитики и проверки рекомендаций (промежуточная версия).  
- **3.0.0** – база продуктов с БЖУ и предрассчитанными дозами.

## 2. Стек технологий (полный, версия 1.0.0)

| Компонент               | Технология / Библиотека                        | Назначение                                                                 |
|-------------------------|------------------------------------------------|----------------------------------------------------------------------------|
| Язык                    | Python 3.12+                                   |                                                                           |
| Telegram Bot API        | aiogram 3.x                                    | Работа с Telegram (сообщения, группы, FSM, middleware)                    |
| **Гибридный OCR**       | **OpenCV + PaddleOCR + EasyOCR + Tesseract**   | Три независимых движка распознавания с предобработкой и сверкой результатов |
| Предобработка фото      | OpenCV (cv2)                                   | Контраст, адаптивная бинаризация, удаление шумов, обрезка области интереса |
| Основной OCR            | PaddleOCR                                      | Наивысшая точность (92‑95%) на цифровых скриншотах, устойчив к артефактам |
| Резервный OCR           | EasyOCR                                        | Альтернатива (точность ~85‑88%), используется для сравнения               |
| Третий OCR              | Tesseract (pytesseract)                        | Дополнительный независимый движок для повышения надёжности (точность ~80‑90% при правильной настройке) |
| Математика формул       | Python (float/Decimal)                         | Расчёт доз инсулина по персональным коэффициентам                         |
| Управление диалогами    | aiogram FSM (MemoryStorage / SQLiteStorage)    | Пошаговые опросы (/setup, опросы после скриншота)                         |
| Планировщик задач       | APScheduler (с персистентностью в БД)          | Напоминания, повторные опросы, события по расписанию                      |
| База данных (локальная) | SQLite + aiosqlite                             | Настройки, журнал, состояния FSM, напоминания, **результаты трёх OCR**    |
| База данных (продакшн)  | PostgreSQL + asyncpg                           | Для версии 2.0.0                                                          |
| Логирование             | loguru                                         | Отладка и мониторинг ошибок                                               |
| Конфигурация            | python-dotenv                                  | Переменные окружения (токен, API‑ключи)                                   |
| Тестирование            | pytest + pytest-asyncio                        | Модульные тесты распознавания, формул, логики согласованности             |

**Почему именно три OCR‑движка:**  
- Обеспечение **отказоустойчивости** – если один сервис даст сбой (редко, но возможно), работа продолжается.  
- **Контроль достоверности** – три результата сравниваются; расхождение >1 единицы сахара сигнализирует о возможной ошибке распознавания и требует ручного ввода.  
- Комбинация OpenCV‑предобработки повышает точность каждого движка на 15‑20% за счёт устранения шумов и выделения области с числом.

## 3. Архитектура бота (обновлённая, с тройным OCR)

```mermaid
flowchart TB
    User[Пользователь] -->|фото/сообщение| TG[Telegram]
    TG --> Dispatcher[aiogram Dispatcher]
    
    subgraph Handlers
        PhotoHandler[Хендлер фото]
        CmdHandler[Хендлер команд]
        FoodHandler[Хендлер фраз про еду]
        FSMHandler[Хендлеры состояний]
    end
    
    subgraph OCR_Engine["Гибридный OCR (OpenCV + 3 движка)"]
        Preprocess[OpenCV: серый, бинаризация, обрезка] --> OCR1[PaddleOCR]
        Preprocess --> OCR2[EasyOCR]
        Preprocess --> OCR3[Tesseract]
        OCR1 --> ResultAgg[Агрегатор результатов]
        OCR2 --> ResultAgg
        OCR3 --> ResultAgg
        ResultAgg -->|проверка согласованности| FinalValue{Надёжное значение?}
        FinalValue -->|да| FormulaService
        FinalValue -->|нет| FailMessage[Сообщение о расхождении<br>+ запрос ручного ввода]
    end
    
    subgraph Services
        FormulaService[Расчёт доз:
            - ISF (утро/день/вечер)
            - I:C ratio
            - Добавки на приёмы
            - Целевой сахар]
        JournalService[Журнал: запись / выборка]
        ReminderService[APScheduler + БД]
        ProfileService[Настройки профиля]
    end
    
    subgraph Database
        DB[(SQLite / PostgreSQL)]
    end
    
    Dispatcher --> Handlers
    PhotoHandler --> OCR_Engine
    CmdHandler --> Services
    FoodHandler --> FormulaService
    FSMHandler --> ProfileService
    Services --> DB
    ReminderService --> Dispatcher
```

**Взаимодействие модулей:**

1. **PhotoHandler** – получает фото, вызывает гибридный OCR.  
2. **OCR_Engine** – три движка параллельно извлекают число, агрегатор сравнивает их. Если согласованность высока – передаёт среднее/медиану в FormulaService. Иначе – отправляет пользователю сообщение с предложением ввести сахар вручную.  
3. **FormulaService** – получает надёжное значение сахара, при необходимости запрашивает тип расчёта (только коррекция или комбинация с едой).  
4. **FoodHandler** – распознаёт запросы на расчёт еды и запускает FormulaService.  
5. **JournalService** – пишет все события, включая **результаты трёх OCR** (три отдельных поля + итоговое значение).  
6. **ReminderService** – работает по расписанию и создаёт динамические напоминания после приёма пищи.

## 4. Сценарии пользователя (глазами пользователя)

### 4.1 Добавление бота в семейную группу
1. Администратор группы добавляет бота `@DiabetFamilyBot` и делает его **администратором** (право «читать сообщения»).  
2. Бот приветствует группу: *«Привет! Я помощник по диабету. Для начала выполните /setup в личных сообщениях или здесь»*.

### 4.2 Первая настройка – профиль диабетика (`/setup`)
Запускается пошаговый опрос (FSM). Параметры привязаны к ID чата, в котором выполнена команда (рекомендуется ЛС).

1. **Единицы измерения** `mmol/L` или `mg/dL`.  
2. **Целевой уровень глюкозы** (например, `5.5`).  
3. **Углеводный коэффициент** `1 ХЕ = X г углеводов` (обычно 10–12).  
4. **Фактор чувствительности к инсулину (ISF)** – на сколько снижает сахар 1 ед. инсулина. Может быть общим или **три значения**:  
   - Утро (например, `2.5`)  
   - День (`3.0`)  
   - Вечер (`2.8`)  
5. **Инсулин/ХЕ коэффициент (I:C ratio)** – сколько единиц короткого инсулина нужно на 1 ХЕ. Может быть общим или разным для завтрака/обеда/ужина.  
6. **Фиксированные добавки** (опционально):  
   - Завтрак: `+2.0` ед.  
   - Обед: `+1.5` ед.  
   - Ужин: `+2.5` ед.  
7. **Длинный инсулин (базальный)**:  
   - Доза, время введения (например, `22:00`), кратность (1 или 2 раза в сутки).  
8. **Напоминания** – стандартные и пользовательские.

После сохранения бот выводит сводку настроек.

### 4.3 Отправка скриншота Libre2 (тройное распознавание)
1. Пользователь делает скриншот и отправляет в группу/боту.  
2. Бот запускает гибридный OCR: предобработка и три движка.  
3. **Если все три результата в пределах ±0.5 mmol/L (±9 mg/dL):**  
   - Итоговое значение = среднее арифметическое (или медиана).  
   - Бот пишет: *«🔍 Распознан сахар: 12.3 mmol/L (подтверждено тремя движками)»*.  
   - Переходит к расчёту коррекции или комбинации (если есть запрос на еду).  
4. **Если два результата близки, а третий отличается ≤1.0 единицы:**  
   - Итоговое = среднее двух близких. Бот уведомляет: *«⚠️ Небольшое расхождение (12.1 и 12.3, третий: 13.0). Принято 12.2»*.  
5. **Если разброс >1.0 единицы или какой-то движок вернул явную ошибку:**  
   - Бот отвечает: *«❌ Показания сильно разнятся: Paddle=11.2, Easy=13.8, Tesseract=9.7. Пожалуйста, введите сахар вручную командой /sugar»*.  
   - Все три значения всё равно сохраняются в журнал с пометкой `unreliable`.

**После получения надёжного сахара:**  
- Бот вычисляет коррекцию (если нет запроса на еду) или использует как составную часть комбинированной дозы (см. п.4.4).

### 4.4 Расчёт дозы на углеводы по фразе
1. Пользователь пишет: *«какой укол 45 23 67»*.  
2. Бот извлекает числа, опционально определяет приём пищи («завтрак» и т.п.).  
3. Суммирует углеводы, переводит в ХЕ, считает дозу на еду.  
4. **Если есть актуальный сахар (последний замер ≤15 мин, надёжный):**  
   - Добавляет коррекцию: `(текущий_сахар - целевой) / ISF_текущего_периода`.  
   - Учитывает добавку на приём пищи.  
5. Выводит полную раскладку:  
   *«📊 Углеводы: 135 г → 13.5 ХЕ (10 г/ХЕ).  
   🍽 Доза на еду: 13.5 × 1.5 = 20.3 ед + завтрак +2.0 = 22.3 ед.  
   🩸 Сахар 12.3 (15 мин) → коррекция (12.3‑5.5)/2.5 = +2.7 ед.  
   💉 Общая рекомендуемая доза: 25.0 ед.»*  

### 4.5 Ручной ввод данных
| Команда                | Пример                        | Действие                                                     |
|------------------------|-------------------------------|--------------------------------------------------------------|
| `/sugar 8.2`           | Запись сахара                 | Сохраняет замер в журнал, может предложить коррекцию         |
| `/carbs 45`            | Углеводы (г)                  | Пересчитывает в ХЕ, запускает расчёт дозы                    |
| `/he 5.5`              | ХЕ напрямую                   | Сохраняет ХЕ и предлагает дозу                               |
| `/insulin short 4.5`   | Укол короткого инсулина       | Записывает факт укола                                        |
| `/insulin long 18`     | Укол длинного                 | Записывает, обновляет напоминание                            |
| `/note Текст заметки`  | Произвольная запись           | Добавляет запись с временем и текстом                        |

### 4.6 Журнал замеров (`/log`)
- `/log day` / `week` / `month` – бот выводит записи за период.  
- При большом объёме формируется CSV‑файл и отправляется как документ.  
- **Для каждого скриншота в журнале сохраняются три значения от OCR‑движков и итоговое**, что позволяет анализировать качество распознавания.

### 4.7 Напоминания
- Стандартные и пользовательские, с персистентностью в БД.  
- Динамические (через 2 часа после еды) создаются автоматически при записи укола короткого + углеводов.

### 4.8 Что видит пользователь при ошибках
- Если все OCR не смогли прочитать число: *«Не удалось распознать сахар. Пожалуйста, введите вручную командой /sugar»*.  
- Если в запросе на еду нет чисел: *«Не найдены углеводы. Напишите числа через пробел»*.  
- Если профиль не настроен: *«Сначала выполните /setup»*.

## 5. Детали реализации ключевых модулей

### 5.1 Гибридный OCR‑модуль (OCRService)
**Цель:** получить максимально достоверное число уровня глюкозы из скриншота Libre2.

**Алгоритм:**
1. **Предобработка (OpenCV):**  
   - Перевод в оттенки серого.  
   - Адаптивная бинаризация (`cv2.adaptiveThreshold`) для усиления контраста цифр.  
   - Обрезка до области, где гарантированно находится значение (задаётся координатами/процентами от размера, либо определяется автоматически).  
2. **Параллельный запуск трёх движков (asyncio.to_thread):**  
   - **PaddleOCR** (основной) – высокая точность.  
   - **EasyOCR** – резервный.  
   - **Tesseract** (pytesseract) – третий независимый, с настроенным `--psm 7` (одна строка текста).  
3. **Извлечение чисел:** из текста каждого движка регулярным выражением ищется первое число с плавающей точкой (ожидаемый формат).  
4. **Агрегация и проверка согласованности:**  
   - Если все три числа получены, вычисляется максимальная разница (`max - min`).  
   - Порог согласованности **1.0 mmol/L** (или эквивалент в mg/dL, ~18).  
   - Если разброс ≤ порога – итоговое значение = медиана (или среднее). Сохраняется флаг `reliable=True`.  
   - Если разброс > порога или хотя бы один движок не вернул число – выставляется `reliable=False`, пользователю предлагается ручной ввод.  
5. **Логирование:** все три результата + итоговое решение записываются в лог.

**Хранение в БД:**  
Таблица `glucose_log` содержит поля:  
- `value_ocr1` (PaddleOCR),  
- `value_ocr2` (EasyOCR),  
- `value_ocr3` (Tesseract),  
- `final_value` (используемое в расчётах),  
- `is_reliable` (bool),  
- `source` = `'ocr'` или `'manual'`.

### 5.2 Модуль формул (FormulaService)
**Параметры (хранятся для каждого chat_id):**
- `units` (mmol / mgdl)  
- `target_glucose`  
- `isf_morning, isf_day, isf_evening` (или единый `isf`)  
- `carb_ratio` (общий или по приёмам)  
- `add_breakfast, add_lunch, add_dinner`  
- `bread_unit_grams` (вес 1 ХЕ в граммах углеводов)  
- `basal_dose, basal_time1, basal_time2`

**Расчёт дозы на еду + коррекцию (при наличии актуального сахара):**  
1. `HE = total_carbs / bread_unit_grams`  
2. `meal_dose = HE * carb_ratio` (если ratio зависит от приёма – выбрать соответствующий).  
3. Добавить `add_meal`, если приём пищи определён.  
4. Если передан `current_glucose` и его возраст ≤15 мин:  
   - `isf` выбрать по времени суток.  
   - `correction = (current_glucose - target_glucose) / isf`.  
   - `total = meal_dose + correction`.  
5. Округление до 0.5 ед. (настраиваемое).  
6. Возврат детализированного ответа.

### 5.3 Обработка запросов на еду (FoodHandler)
- Триггеры: фразы `какой укол`, `сколько укола`, `посчитай укол`.  
- Извлекает все числа из сообщения и определяет приём пищи по ключевым словам.  
- Вызывает FormulaService.  
- Если нет настроек – предлагает `/setup`.  
- Если есть сохранённый последний надёжный сахар ≤15 мин – использует его.

### 5.4 FSM и опросы
- Реализованы через aiogram FSM.  
- Состояния: `ProfileSetup`, `InjectionConfirm`, `ReminderSetup`, `LogPeriod`.  
- Хранилище в версии 1.0.0 – MemoryStorage; для 2.0.0 – Redis.

### 5.5 База данных (SQLite)
**Основные таблицы:**
- `chats` – настройки профиля.  
- `glucose_log` – (id, chat_id, user_id, timestamp, value_ocr1, value_ocr2, value_ocr3, final_value, is_reliable, source).  
- `food_log` – углеводы, ХЕ, тип приёма.  
- `insulin_log` – уколы (тип, доза).  
- `notes` – произвольные заметки.  
- `reminders` – активные напоминания.  

### 5.6 Планировщик (ReminderService)
- Использует APScheduler с SQLiteJobStore для сохранения задач между перезапусками.  
- При старте бот загружает все активные напоминания.  
- Динамические напоминания (через 2 часа после еды) создаются как одноразовые задачи и удаляются после выполнения.

## 6. Процесс разработки и отладки (на домашнем ПК, версия 1.0.0)

### 6.1 Установка окружения
```bash
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install aiogram[redis,sqlite] opencv-python paddlepaddle paddleocr easyocr pytesseract aiosqlite apscheduler loguru python-dotenv
```

| Чего избегать                                     | Почему                                                        | Правильный подход                                   |
|---------------------------------------------------|---------------------------------------------------------------|-----------------------------------------------------|
| Хранение состояний в глобальных переменных        | Потеря при перезапуске                                        | FSM Storage (Memory/Redis)                          |

## 10. Заключение

Бот построен на принципах **надёжности, прозрачности и модульности**.  
Гибридный OCR с тремя независимыми движками обеспечивает высокую достоверность распознавания, а автоматическая проверка согласованности исключает скрытые ошибки. Пользователь всегда видит, какое значение взято в расчёт, и может скорректировать его вручную.  
Полноценный расчётный движок учитывает все нюансы инсулинотерапии (разные коэффициенты по времени суток, добавки на приёмы пищи, коррекцию).  
Ведение детального журнала (включая три значения OCR) создаёт основу для аналитики и будущей интеграции с ИИ‑ассистентом (DeepSeek).  
Модульная архитектура позволяет развивать бота поэтапно, заменяя компоненты без переписывания всего кода.
