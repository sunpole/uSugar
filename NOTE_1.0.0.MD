# Техническая документация: Telegram-бот для семейного мониторинга сахара (Libre2)

## 1. Общее описание

Бот предназначен для семейной группы (родители+ребёнок с диабетом). Он:
- распознаёт скриншоты из приложения Libre2 (уровень глюкозы в mmol/L);
- по заданным формулам рассчитывает дозу инсулина или даёт рекомендации;
- может опрашивать пользователя о сделанных процедурах (укол, еда) и ждать ответа;
- отправляет напоминания по расписанию;
- работает как в личных сообщениях, так и в групповом чате (требуются права администратора).

**Версия 1.0.0** – запуск на домашнем ПК (разработка и отладка).  
**Версия 2.0.0** – деплой на выделенном хостинге (работа 24/7).

## 2. Стек технологий (полный)

| Компонент             | Технология / Библиотека          | Назначение                                                                 |
|-----------------------|----------------------------------|----------------------------------------------------------------------------|
| Язык                  | Python 3.12+                     |                                                                           |
| Telegram Bot API      | aiogram 3.x                      | Работа с Telegram (сообщения, группы, FSM, middleware)                    |
| OCR (распознавание)   | PaddleOCR (основной) + EasyOCR   | Извлечение числа глюкозы из скриншота. PaddleOCR точнее, EasyOCR – резерв |
| Предобработка фото    | OpenCV (cv2)                     | Контраст, бинаризация, удаление шумов перед OCR                           |
| Математика формул     | Python (math, decimal)           | Расчёт инсулина по заданным коэффициентам                                 |
| Управление диалогами  | aiogram FSM (State, Context)     | Пошаговые опросы: ожидание ответа “сделал укол?” и т.п.                   |
| Планировщик задач     | APScheduler                      | Напоминания (например, “через 2 часа после еды”)                          |
| База данных (локальная) | SQLite (aiosqlite)             | Хранение настроек, протоколов, истории, состояний FSM                     |
| База данных (продакшн) | PostgreSQL + asyncpg            | Для версии 2.0.0 на хостинге                                               |
| Логирование            | loguru                           | Отладка и мониторинг ошибок                                               |
| Конфигурация           | pydantic-settings (.env)         | Переменные окружения (токены, формулы, ID чатов)                          |
| Тестирование           | pytest + pytest-asyncio          | Модульные тесты распознавания и формул                                    |

### Почему выбор пал на PaddleOCR вместо чистого EasyOCR?
- PaddleOCR даёт точность 92–95% на цифрах и коротком тексте, EasyOCR ~85–88%.
- Скриншоты Libre2 имеют мелкий шрифт и артефакты сжатия – PaddleOCR устойчивее.
- EasyOCR оставлен как fallback.

## 3. Архитектура бота (графически + текстом)

```mermaid
flowchart TB
    User[Пользователь в группе/лс] -->|отправляет фото| TG[Telegram]
    TG -->|webhook/polling| Bot[aiogram Dispatcher]
    
    subgraph BotCore
        direction LR
        HandlerPhoto[Хендлер фото] --> Preprocess[OpenCV предобработка]
        Preprocess --> OCR[PaddleOCR / EasyOCR]
        OCR --> ExtractNumber[Извлечение числа по regex]
        ExtractNumber --> FormulaEngine[Движок формул]
        FormulaEngine --> SendResult[Отправка результата в чат]
    end
    
    subgraph FSM_Engine
        StateMachine[FSM Storage<br>SQLite/Redis] --> DialogStep[Шаг опроса]
        DialogStep --> WaitAnswer[Ожидание ответа]
        WaitAnswer --> ProcessAnswer[Обработка ответа]
        ProcessAnswer -->|следующий шаг| DialogStep
        ProcessAnswer -->|завершён| SendResult
    end
    
    subgraph Scheduler
        APScheduler --> CheckTime[Проверка расписания]
        CheckTime --> SendReminder[Отправка напоминания]
    end
    
    subgraph Database
        SQLite[(SQLite / PostgreSQL)]
        Bot -->|чтение/запись| Database
    end
    
    User -->|команда /start /status| Bot
    Bot --> User
```

**Взаимодействие модулей:**
1. **Хендлер фото** – получает медиа, скачивает, передаёт в `preprocessing`.
2. **OCR-модуль** – вызывает PaddleOCR, если не удалось – EasyOCR.
3. **Движок формул** – берёт полученную глюкозу, подставляет в формулу (например, `(глюкоза - 5) / 2`). Формула вычитывается из БД (можно менять через админ-команды).
4. **FSM** – используется для опросов: “Сделал ли ты укол? (да/нет/через 10 минут)”. Состояние висит в БД, бот не теряет диалог даже после перезапуска.
5. **Планировщик** – по расписанию из БД отправляет сообщения (например, каждый день в 21:00: “Проверь сахар перед сном”).
6. **База данных** – хранит:
   - пользователей (user_id, роль, имя);
   - настройки группы (chat_id, активные опросы);
   - протоколы (время, значение сахара, введённый инсулин);
   - расписания напоминаний.

## 4. Сценарии пользователя (глазами пользователя)

### 4.1 Добавление бота в семейную группу
1. **Я** (админ группы) ищу бота по `@DiabetFamilyBot` в Telegram.
2. Нажимаю **“Добавить в группу”**, выбираю нашу семейную группу.
3. **Бот** просит меня сделать его администратором (нужны права “просмотр сообщений”).
4. **Я** захожу в настройки группы → “Управление” → “Администраторы” → добавляю бота с галочкой “читать все сообщения”.
5. **Бот** приветствует: *“Привет, я буду помогать с диабетом. Используй /setup, чтобы настроить формулы и напоминания”*.

### 4.2 Первая настройка (команда /setup в личке с ботом)
1. **Я** пишу `/setup` боту в личный чат.
2. **Бот** спрашивает: *“Введите формулу для расчёта дозы (например, (value-5)/2). value – это уровень сахара”*
   - **Я** пишу: `(value - 4.5) / 1.8`
   - **Бот** запоминает формулу для этого чата.
3. **Бот**: *“Хотите настроить напоминания? Напишите время и текст (например, 21:00 Проверь сахар)”*
   - **Я** отправляю: `21:00 Измерь сахар перед сном`
   - **Бот** сохраняет и говорит *“Напоминание установлено”*.

### 4.3 Использование в группе: отправка скриншота
1. **Ребёнок (или я)** делает скриншот в приложении Libre2.
2. Отправляет скриншот в семейную группу, можно с комментарием.
3. **Бот** видит фото (благодаря правам администратора). Распознаёт значение `23.1`.
4. **Бот** пишет в чат: *“🔍 Распознан сахар: 23.1 mmol/L. По формуле (value-4.5)/1.8 = (23.1-4.5)/1.8 = 10.3 единиц. Рекомендуемый укол – 10.3 ед.”*
5. **Бот** задаёт вопрос (FSM): *“Сделали укол? (да / пропустить)”*
6. **Я** нажимаю “да”. **Бот** отвечает: *“✅ Отлично. Через 2 часа напомню проверить сахар.”*
   - Планировщик ставит задачу через 2 часа.

### 4.4 Напоминание о проверке
1. Через 2 часа **бот** пишет в группе: *“⏰ Прошло 2 часа после укола. Отправьте текущий сахар.”*
2. **Я** отправляю новый скриншот. Бот распознаёт, считает, и цикл повторяется.

### 4.5 Если ответ на вопрос не пришёл
- **Бот** через 5 минут повторяет: *“Вы не ответили. Укол сделан? (да/нет)”* – всего до 3 раз, потом снимает опрос.

### 4.6 Что видит пользователь при ошибке
- Если OCR не распознал число: *“Не удалось прочитать сахар. Попробуйте фото чётче или введите вручную: /sugar 23.1”*

## 5. Процесс разработки и отладки (на домашнем ПК, версия 1.0.0)

### 5.1 Установка окружения (Windows/Mac/Linux)
```bash
python -m venv venv
source venv/bin/activate  # или venv\Scripts\activate
pip install aiogram==3.10.0 opencv-python paddlepaddle paddleocr easyocr aiosqlite apscheduler loguru python-dotenv
```

### 5.2 Структура проекта (модульная)

```
diabet_bot/
├── bot.py                 # точка входа, запуск polling
├── config.py              # чтение .env (TOKEN, FORMULA)
├── handlers/
│   ├── photo.py           # хендлер фото (OCR + формула)
│   ├── commands.py        # /start, /setup, /status
│   └── fsm.py             # опросы (укол, еда)
├── services/
│   ├── ocr_service.py     # PaddleOCR + EasyOCR + OpenCV
│   ├── formula_service.py # вычисление по формуле из БД
│   └── reminder_service.py# APScheduler
├── db/
│   ├── models.py          # SQLAlchemy или сырые SQL-запросы
│   └── database.py        # инициализация SQLite
├── states/
│   └── user_states.py     # классы FSM (InjectionState, MealState)
├── utils/
│   ├── logger.py          # loguru
│   └── helpers.py         # regex, валидация
└── .env                   # TOKEN, FORMULA, CHAT_ID
```

### 5.3 Отладка по шагам на ПК
1. Создаю бота у `@BotFather`, получаю токен, записываю в `.env`.
2. Запускаю `python bot.py`. Бот работает через long polling.
3. Добавляю бота в тестовую группу (или пишу в личку).
4. Отправляю тестовый скриншот – смотрю в консоли лог (loguru).
5. Проверяю, корректно ли распозналось число. Если нет – регулирую OpenCV (пороги бинаризации).
6. Проверяю FSM: отправляю команду, имитирую ответы, наблюдаю состояния в БД (`sqlite3 diabet.db`).

**Сложности на этом этапе:**
- PaddleOCR требует загрузки моделей (~15-20 МБ) при первом запуске. На ПК это нормально.
- На Windows могут быть проблемы с установкой PaddlePaddle – лучше использовать WSL2 или Docker.

### 5.4 Что проверяется перед переходом на хостинг
- [ ] Распознавание 5 разных скриншотов Libre2 (разное освещение, повороты).
- [ ] Корректная работа формул с плавающей точкой.
- [ ] FSM не зависает при нескольких одновременных опросах (разные пользователи).
- [ ] Планировщик отправляет напоминания в нужное время (проверяем, меняя системное время).
- [ ] Бот корректно работает в группе как администратор.

## 7. Самые сложные места и как их избежать

### 7.1 Точность распознавания
**Проблема:** Скриншоты могут быть разного качества, с отражениями, с цифрами на фоне графика.  
**Решение:** 
- Предобработка OpenCV: преобразование в оттенки серого, адаптивная бинаризация (`cv2.adaptiveThreshold`).
- Обрезать область экрана, где точно находится число (можно жёстко задать координаты для Libre2).  
- Добавить ручной ввод `/sugar 23.1` как fallback.

### 7.2 Состояния FSM и перезагрузка бота
**Проблема:** При перезапуске бота (например, обновление кода) состояния пользователей теряются, если хранить в оперативной памяти.  
**Решение:** Использовать FSM Storage на основе SQLite/PostgreSQL. В aiogram это делается через `RedisStorage2` или `SQLiteStorage`. Для 1.0 подойдёт `MemoryStorage` (при перезапуске всё сбросится), для 2.0 – `Redis`.

### 7.3 Напоминания при отключённом боте
**Проблема:** Если бот упадёт на 2 часа, запланированные события не сработают.  
**Решение:** 
- APScheduler хранит задачи в памяти, при перезапуске они теряются. Нужно при старте бота загружать активные напоминания из БД и пересоздавать их.  
- Либо использовать `apscheduler.sqlalchemy` для персистентности.

### 7.4 Работа в группе – конфликт команд
**Проблема:** Бот реагирует на каждое фото, даже если это не скриншот Libre2.  
**Решение:** 
- Проверять наличие характерных элементов Libre2 на фото (например, зелёная кривая графика) перед запуском OCR. Это снижает ложные срабатывания.
- Использовать фильтр на упоминание бота: бот обрабатывает фото только если в сообщении есть `@DiabetFamilyBot` или если фото отправлено в ответ на вопрос бота.

### 7.5 Многопользовательские опросы
**Проблема:** В группе одновременно несколько людей могут начать диалог с ботом (например, один отвечает на вопрос про укол, другой просит формулу).  
**Решение:** 
- Привязывать состояние FSM к `(chat_id, user_id)`. Aiogram это поддерживает из коробки.  
- В сообщениях бота упоминать пользователя через `reply_to_message` или `@username`.

## 8. Модульность проекта – насколько нужна?

**Коротко: жизненно необходима.**  
Бот с таким функционалом без модульной архитектуры через месяц превратится в спагетти-код, который невозможно отлаживать.

### Что строго должно быть в отдельных модулях:
- **OCR service** – изолированная логика распознавания. Можете заменить PaddleOCR на Google Vision без правки остального кода.
- **Formula service** – загрузка формулы из БД, вычисление. Позволяет менять формулу на лету (через админ-команду).
- **Database layer** – все запросы к БД в одном месте, не размазаны по хендлерам.
- **State definitions** – все FSM-классы в отдельном файле, легко добавить новый опрос.
- **Middleware** – единая обработка ошибок, логирование, проверка прав.

### Признак, что модульность достаточная:
- Вы можете протестировать OCR-функцию, не запуская бота целиком.
- Вы можете сменить БД с SQLite на PostgreSQL, изменив ровно один файл.

## 9. Чего избегать при создании бота

| Чего избегать                          | Почему                                                        | Правильная альтернатива                          |
|----------------------------------------|---------------------------------------------------------------|--------------------------------------------------|
| Хранить состояния в глобальных переменных | Потеря состояний при любом перезапуске                        | FSM Storage на БД или Redis                     |
| Обрабатывать фото синхронно (долгий OCR) | Бот будет зависать для всех остальных пользователей            | Использовать `asyncio.to_thread` для OCR        |
| Задавать вопросы без таймаута           | Бот будет ждать ответ вечно, забивая память                   | Таймаут 5-10 минут, потом отмена опроса         |
| Игнорировать ошибки OCR                 | Пользователь не поймёт, почему бот молчит                     | Явное сообщение “Не распознано, попробуйте ещё” |
| Твёрдо зашивать формулу в коде           | Чтобы изменить формулу, нужно перезапускать бота              | Хранить в БД и перечитывать при каждом расчёте (или с кэшем) |
| Деплоить без тестов                     | С вероятностью 90% что-то сломается в первый же день в группе  | Написать хотя бы 5-10 тестов на критичные функции |

- Стек: **Python 3.12+, aiogram 3.x, PaddleOCR+EasyOCR, OpenCV, APScheduler, SQLite/PostgreSQL**.
- Архитектура: событийная, модульная, с выделением сервисов OCR, формул и планировщика.
