# VERSION_DIAGNOSTICS - uSugar

Дата диагностики: 2026-06-11

## Краткий вывод

В документации и на диске проект уже приведён к `1.1.0`, но живой Telegram-бот отвечает как `1.0.39`, потому что запущен старый runtime-процесс, который не был перезапущен после обновления версии.

## 1. Где хранится номер версии

Источник версии сейчас разделён на несколько уровней:

1. `VERSION.json` - главный источник истины для runtime-версии.
2. `version_info.py` - читает `VERSION.json` и экспортирует `VERSION`.
3. `config.py` - импортирует `VERSION` из `version_info.py`.
4. `bot.py` - использует `VERSION` из `config.py` для `/version`, `/health`, `/help`, OCR-текстов, reminders и startup banner.
5. `settings.html` - получает версию через `update_version_in_html()` из `bot.py`.

## 2. Какие файлы содержат `1.0.39`

Исторически и/или явно `1.0.39` встречается в:

- `VERSION.json` - в `history[]` и в notes про bump до `1.1.0`;
- `CHANGELOG.md` - секция `1.0.39`;
- `PROJECT_STATUS.md` - упоминание story landing `1.0.39`;
- `ROADMAP.md` - ссылка на старт story landing в `1.0.39`;
- `PROJECT_AUDIT.md` - текущая версия в этом аудите;
- `CODE_AUDIT.md` - текущая версия в этом аудите;
- `PROJECT_STORY_AUDIT.md` - историческая запись про story landing `1.0.39`;
- `docs/history/SCREENSHOT_STORY.md` - запись про `1.0.39_story_landing`;
- `docs/history/story_data.js` - caption/version для `1.0.39`.

Это не ошибка само по себе: часть этих файлов хранит историю релиза, а не только текущую версию.

## 3. Какие файлы содержат `1.1.0`

Актуальная версия `1.1.0` уже есть в:

- `VERSION.json` - текущий top-level `version`;
- `README.md` - current local code version;
- `PROJECT_STATUS.md` - current working code identifies itself as `1.1.0`;
- `CHANGELOG.md` - секция `1.1.0`;
- `ROADMAP.md` - пункты, отмеченные как done in `1.1.0`;
- `FUTURE_BACKLOG.md` - deferred work after `1.1.0`;
- `docs/AI_CONTEXT.md` - текущая версия проекта по активной документации;
- `settings.html` - отображаемая версия `uSugar v1.1.0`.

## 4. Откуда команда `/version` берёт значение

Цепочка такая:

`/version` в `bot.py` -> `VERSION` из `config.py` -> `VERSION` из `version_info.py` -> `VERSION.json`

В `bot.py` обработчик `/version` напрямую отвечает строкой вида:

`uSugarBot v{VERSION}`

Поэтому если Telegram показывает `uSugarBot v1.0.39`, это означает, что в памяти живого процесса `VERSION` ещё равен `1.0.39`.

## 5. Какой процесс сейчас запущен

По состоянию на диагностику активен bot runtime:

- PID: `12644`
- команда запуска: `"C:\!CODE_CLUB\new 2026\002_usugar\venv\Scripts\python.exe" -u bot.py`
- PID-файл: `.runtime\bot.pid` содержит тот же PID `12644`
- startup log: `.runtime\logs\bot.out.log` содержит строку `uSugarBot v1.0.39 запущен и готов к работе!`
- время старта в логе: `2026-06-11 07:57:47`

Также виден связанный процесс:

- PID: `5380`
- executable path: `C:\Program Files\WindowsApps\PythonSoftwareFoundation.Python.3.9_3.9.3568.0_x64__qbz5n2kfra8p0\python3.9.exe`
- command line: `python.exe -u bot.py`

Но ключевая точка для Telegram - именно PID `12644`.

## 6. Не используется ли старая копия проекта

По доступным признакам самая вероятная причина не в отдельной старой папке, а в том, что:

- бот был запущен до обновления версии до `1.1.0`;
- процесс продолжает работать без перезапуска;
- runtime banner в `bot.out.log` всё ещё говорит `1.0.39`.

То есть живой Telegram-бот, скорее всего, использует старый runtime-снимок в памяти, а не свежий запуск.

При этом `ExecutablePath` у активного процесса указывает на текущий локальный venv:

`C:\!CODE_CLUB\new 2026\002_usugar\venv\Scripts\python.exe`

Это делает версию "другая копия проекта" менее вероятной, чем версию "старый процесс не перезапущен".

## 7. Не использует ли Telegram старый runtime

Да, по наблюдаемым данным Telegram сейчас видит старый runtime:

- в Telegram `/version` показывает `uSugarBot v1.0.39`;
- `bot.out.log` подтверждает запуск именно как `v1.0.39`;
- на диске уже лежит актуальная документация и версия `1.1.0`;
- значит, Telegram общается с тем процессом, который был поднят раньше и не был перезапущен.

## 8. Источник проблемы

Источник проблемы: старый долгоживущий процесс `bot.py`, запущенный до версии `1.1.0`, продолжает обслуживать Telegram. Версия на диске уже обновлена, но runtime не перечитал её заново, потому что Python загрузил `VERSION.json` только при старте процесса.

## 9. Минимальное исправление

Самое маленькое безопасное исправление:

1. Остановить текущий bot runtime PID `12644`.
2. Запустить `bot.py` заново из текущего рабочего каталога `C:\!CODE_CLUB\new 2026\002_usugar`.
3. Сразу проверить `/version` в Telegram.

Если после перезапуска Telegram всё ещё покажет `1.0.39`, тогда нужно проверить запуск из другой копии или из другого ярлыка/скрипта. Но по текущим данным это менее вероятно.

## 10. Нужно ли просто перезапустить бота

Да. По симптомам этого достаточно в первую очередь.

Сильный признак именно stale runtime:

- на диске уже `1.1.0`;
- в логе старт всё ещё `1.0.39`;
- PID-файл и активный процесс указывают на давно живущий bot runtime.

## 11. Какие файлы должны содержать актуальную версию

Для синхронного и понятного состояния актуальную версию `1.1.0` должны содержать:

- `VERSION.json` - всегда;
- `version_info.py` - через чтение `VERSION.json`, а не через hardcode;
- `config.py` - только как импортированный `VERSION`;
- `bot.py` - только как runtime consumer;
- `settings.html` - отображаемый version badge;
- `README.md` - current local version;
- `PROJECT_STATUS.md` - current state;
- `CHANGELOG.md` - секция `1.1.0`;
- `ROADMAP.md` - ссылки на этапы;
- `FUTURE_BACKLOG.md` - отложенный после `1.1.0` scope;
- `docs/AI_CONTEXT.md` - текущий навигационный статус;
- `docs/history/story_data.js` и `docs/history/SCREENSHOT_STORY.md` - только если там речь идёт о новом релизном контексте, а исторические `1.0.39` записи должны оставаться как история.

## 12. Практический итог

Проблема не в том, что где-то "неправильная версия в коде". Проблема в том, что Telegram разговаривает со старым работающим процессом. Минимальный ремонт - перезапуск бота. После этого `/version` должен начать показывать `uSugarBot v1.1.0`, если запуск идёт из этой же актуальной папки.
